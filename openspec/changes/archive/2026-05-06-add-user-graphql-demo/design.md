## Context

当前仓库由独立运行的 NestJS 后端与 Vite React 前端组成，尚未具备真实的数据模块和 GraphQL 集成示例。后端当前只有单一控制器和服务，前端当前只有一个单页界面，因此本次变更需要同时补充后端模块化结构、GraphQL 入口以及前端调用示例，并保持实现尽量轻量，便于作为后续开发模板。

## Goals / Non-Goals

**Goals:**
- 在后端引入最小可用的 GraphQL 服务，并提供用户查询示例。
- 在后端通过独立 `user` 模块组织用户领域的 resolver、service 和数据定义。
- 在前端提供一个基于 Apollo Client 的 GraphQL demo 独立页面。
- 在前端引入最小路由系统，承载 GraphQL demo 页面入口。
- 保持前后端在当前独立应用结构下可本地联调，不引入额外的复杂基础设施。

**Non-Goals:**
- 不实现用户注册、登录、权限控制或持久化数据库。

## Decisions

1. 采用 NestJS GraphQL code-first 模式。
   - 原因：当前后端规模很小，code-first 更适合直接从 TypeScript 类定义 schema，减少额外 SDL 文件维护成本。
   - 备选方案：schema-first 可以更明确管理 GraphQL SDL，但会增加当前示例项目的文件与维护负担。

2. 用户数据先使用内存静态数据源。
   - 原因：本次目标是建立模块、接口和前端调用链路，而不是引入数据库迁移与持久化复杂度。
   - 备选方案：接入 SQLite 或其他轻量数据库能更贴近真实场景，但会显著扩大实现范围。

3. GraphQL API 提供只读查询能力，至少包含查询全部用户与按 id 查询单个用户。
   - 原因：查询场景足以演示 schema、resolver、service 协作以及前端调用方式。
   - 备选方案：同时提供 mutation 能展示更完整能力，但不属于当前最小演示范围。

4. 前端使用 Apollo Client 调用 GraphQL HTTP 端点。
   - 原因：Goals 已明确要求引入 Apollo Client，且它能统一管理 GraphQL 请求、缓存与错误状态，便于后续扩展。
   - 备选方案：直接使用 `fetch` 实现更轻量，但不符合当前变更目标；其他客户端库也缺少 Apollo 在 React 生态中的常见示例价值。

5. 前端增加最小路由系统，将 GraphQL demo 作为独立页面挂载。
   - 原因：Goals 已明确要求新增路由系统，独立页面入口比把 demo 继续堆叠在单页中更清晰。
   - 备选方案：继续在单一 `App.tsx` 中直接渲染 demo 实现更简单，但不满足当前范围。

6. 通过后端启用宽松的本地 CORS 支持前端开发访问。
   - 原因：前后端是独立开发服务器，前端 demo 需要直接访问本地 GraphQL 接口。
   - 备选方案：使用 Vite 代理也可行，但后端启用 CORS 更直接且不依赖前端特定配置。

## Risks / Trade-offs

- [内存数据无法覆盖真实持久化场景] → 通过在 spec 中明确该能力是演示级查询接口，后续可在 service 层替换数据源。
- [前后端独立启动增加本地联调步骤] → 在任务中包含最小联调与验证步骤，确保演示链路清晰。
- [引入 GraphQL 依赖会改变后端启动配置] → 将变更限定在最小必要配置，避免影响现有 REST 示例接口。
