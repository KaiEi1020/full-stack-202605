## Context

当前 `backend/` 仍采用 NestJS 默认 controller/service 组织方式，用户能力基于内存数据实现，缺少稳定的数据持久化层与明确的领域边界。现有仓库已经引入基础 GraphQL 查询示例，但后端能力仍偏向 demo 级实现，不足以支撑真实业务扩展、数据演进与测试分层。

这次改造需要同时处理三类变化：一是引入 Prisma 作为数据库访问与 schema 管理入口；二是按 DDD 方式重组后端模块边界；三是让 GraphQL 成为用户查询能力的主要接口层。由于这三个方向会同时影响目录结构、依赖注入、测试方式与调用链，因此需要先做架构设计，再进入实现。

## Goals / Non-Goals

**Goals:**
- 在 `backend/` 建立 Prisma 基础设施，包括 schema、client 生命周期与数据库访问入口
- 在后端建立 Prisma 基础设施，并支持 DDD 分层与普通 service 两种业务实现模式
- 让 GraphQL resolver 根据业务复杂度，接入应用层/领域层或普通 service 层访问用户数据
- 为后端后续新增能力提供一套可按复杂度选择的 Prisma 架构模式
- 保持系统在开发阶段可本地运行并具备可验证的测试入口

**Non-Goals:**
- 本次不覆盖完整鉴权、授权与多租户模型
- 本次不要求一次性把所有 REST 接口全部迁移为 GraphQL
- 本次不定义生产数据库选型、部署流水线或云端迁移细节
- 本次不扩展复杂写操作，如用户注册、批量导入或事务编排

## Decisions

### 1. 使用 Prisma 作为唯一 ORM 与数据库 schema 事实来源
采用 Prisma schema 描述持久化模型，通过 Prisma Client 访问数据库，并通过 migration 或 db push 管理开发阶段 schema 演进。这样可以避免手写 SQL/Repository 映射分散在各处，也能为测试和后续实体演进提供统一入口。

备选方案：继续使用内存数据或直接引入 TypeORM。前者无法支撑真实持久化需求；后者虽与 Nest 集成成熟，但在 schema 演进、类型推导与当前团队想要的 Prisma 方向上不如 Prisma 直接。

### 2. 同时支持 DDD 分层结构与普通 service 结构
后端按能力域组织，但不强制所有模块都走 DDD。复杂业务场景采用 `domain`、`application`、`infrastructure`、`interfaces` 分层；简单 CRUD 场景允许保留或新增普通 Nest `service` 驱动实现，只要求其底层仍通过 Prisma 访问数据。这样可以把 DDD 留给真正复杂的领域问题，同时避免简单查询和维护型接口过度设计。

备选方案：所有模块统一强制采用 DDD。该方式一致性更强，但会把简单 CRUD 也推入高抽象结构，增加实现与维护成本。

### 3. GraphQL 作为用户读取能力的主要对外接口，保留渐进迁移空间
用户读取场景通过 GraphQL query 暴露，resolver 只负责协议适配和输入输出映射，不承载业务规则。现有 `GET /` 等基础接口可在迁移期暂时保留，但新设计以 GraphQL 为主要读取入口。这样既满足前端按需查询，也避免把 DDD 改造与 REST 全量替换强绑定。

备选方案：保留 REST 为主、GraphQL 仅作 demo。该方案无法体现这次改造的目标，也会让领域层同时服务两套主接口而缺少明确方向。

### 4. 先落地用户查询聚合，作为 Prisma + DDD + GraphQL 的样板
首个聚合选择现有用户能力，只覆盖列表查询与按 id 查询，必要时增加最小字段集和映射规则。这样可在有限范围内验证仓储契约、实体映射、resolver 调用链与测试方式，降低一次性改造全部业务的风险。

备选方案：同时引入多个聚合或直接设计完整用户管理子系统。那会显著扩大范围，拖慢首轮落地。

### 5. 测试按层分离：领域/应用单测 + GraphQL 集成测试
领域层与应用层采用单元测试验证规则和用例编排；GraphQL 入口采用集成或 e2e 测试验证 schema、resolver 与依赖注入链路；Prisma 相关测试优先使用测试数据库或可控数据源，而不是继续依赖纯内存假数据覆盖全部场景。

备选方案：只保留 resolver 级测试。这样无法验证领域与应用层边界，也不利于后续复杂用例扩展。

## Risks / Trade-offs

- [Prisma 引入后需要数据库环境] → 先支持本地开发数据库与最小 schema，降低首次接入门槛
- [DDD 分层会增加初期文件数量与抽象成本] → 先在用户聚合内落地最小必要分层，不预建与当前需求无关的抽象
- [GraphQL 与现有示例实现之间可能出现命名或路径重构] → 以 schema 和领域能力为准统一命名，并通过测试锁定行为
- [团队若尚未统一 DDD 约定，后续实现可能风格漂移] → 在 design 与 tasks 中明确目录结构、依赖方向与职责边界
- [主规格当前按领域组织尚未完全收敛] → 本次 proposal 先按能力拆分产出 specs，实施前应确认是否需要把技术能力继续折叠回领域主规格
