## Context

当前后端用户模块只提供查询能力，前端也只有 GraphQL 查询演示页面，没有可落地的注册流程。此次变更会同时影响 Prisma 数据模型、后端 GraphQL 写接口、消息通知边界以及前端页面交互，属于跨模块改动；同时 RabbitMQ 只作为消息边界引入，短信发送先采用打印占位实现。

## Goals / Non-Goals

**Goals:**
- 在后端新增用户注册能力，接收用户名和手机号并创建用户。
- 让注册流程在成功写入后触发一条 RabbitMQ 消息。
- 提供短信发送占位消费者或处理器，以打印方式代替真实短信发送。
- 在前端新增注册页面并完成表单提交与结果反馈。
- 保持现有用户查询能力可继续工作。

**Non-Goals:**
- 不接入真实短信服务商。
- 不实现登录、鉴权、验证码、风控或手机号唯一性以外的复杂校验体系。
- 不引入完整的消息重试、死信队列或生产级 RabbitMQ 运维方案。
- 不重构当前前端整体路由结构到复杂多页架构。

## Decisions

### 1. 后端继续沿用当前用户模块的分层结构
- 决策：注册能力继续落在 `user` 模块内，保持 resolver -> 应用服务 -> 仓储/基础设施 的调用链。
- 原因：当前查询链路已经采用这一结构，新增写能力时复用同一模块边界最小且一致。
- 备选方案：新增独立注册模块；未采用，因为会把同一聚合的读写能力拆散，当前规模下收益不高。

### 2. 注册接口采用 GraphQL mutation
- 决策：在现有 GraphQL API 中新增 `registerUser` mutation，输入至少包含 `name` 和 `phone`。
- 原因：当前前后端已存在 GraphQL 接口与前端调用模式，继续使用同一协议改动最小。
- 备选方案：新增 REST 接口；未采用，因为会引入第二套协议和文档维护成本。

### 3. 用户持久化模型补齐手机号字段
- 决策：在 Prisma 用户模型中增加手机号字段，并由注册流程写入；如需要，可对手机号建立唯一约束。
- 原因：手机号是注册必填项，必须进入持久化层与 GraphQL 输出模型，才能支持后续通知和去重。
- 备选方案：将手机号只留在消息负载中不入库；未采用，因为无法支持用户数据完整性与后续业务扩展。

### 4. RabbitMQ 作为消息边界，短信发送先用打印替代
- 决策：注册成功后由应用层或消息发布服务发送“用户已注册”消息到 RabbitMQ；消息消费侧调用本地打印方法模拟短信发送。
- 原因：用户明确要求保留 RabbitMQ 形态但无需真实短信，消息边界应先落地，外部集成可后续替换。
- 备选方案：直接在注册成功后同步打印；未采用，因为这无法体现异步消息链路，也不满足 RabbitMQ 约束。

### 5. 前端新增独立注册页面并复用现有单页结构
- 决策：在当前前端中新增注册页面组件与 mutation 调用逻辑，并通过现有 `App.tsx` 入口呈现。
- 原因：当前前端没有路由系统，新增独立页面组件后由 `App.tsx` 承载是最小变更。
- 备选方案：先引入路由；未采用，因为超出当前需求。

## Risks / Trade-offs

- [RabbitMQ 在本地未运行] → 提供可替换的基础设施封装，至少保证打印占位逻辑可验证，并在设计中明确本地依赖。
- [手机号约束不明确导致重复数据] → 在实现时明确唯一性策略，至少在规格中要求系统处理重复手机号注册失败场景。
- [前端当前无路由，注册页与现有演示页并存方式有限] → 使用轻量页面切换或直接替换当前主视图，避免引入额外框架。
- [消息发送成功但短信占位消费失败] → 当前阶段接受为非生产级实现，先确保消息链路和占位处理边界清晰。
