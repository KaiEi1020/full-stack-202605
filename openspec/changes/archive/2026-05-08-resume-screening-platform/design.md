## Context

当前仓库前端只有单页 `App.tsx`，后端只有基础 NestJS 示例接口，尚未形成招聘筛选业务闭环。本次变更需要同时引入文件上传、PDF 解析、AI 结构化提取、SSE 流式返回、岗位评分、候选人管理和较完整的桌面端 UI，因此属于跨前后端的组合式交付。

约束如下：
- 仓库为 pnpm workspace，前端为 Vite + React 19，后端为 NestJS 11。
- 数据必须使用 SQLite + Prisma 持久化，不能使用内存态仓库。
- 原始 PDF 文件直接通过 Node.js `fs` 模块存放在仓库根目录下的专用目录中，由数据库记录文件路径和元数据。
- 大模型能力使用智谱 BigModel HTTP API，API Key 放入环境变量，后端负责发起请求。
- 上传范围调整为单个 PDF 文件，不需要多文件批量上传。
- 目标优先级是“可演示的端到端业务原型”，而不是生产级高并发系统。

## Goals / Non-Goals

**Goals:**
- 构建一个新的招聘筛选工作台页面，覆盖上传、解析、提取、评分、管理四条主链路。
- 为前端提供清晰的组件化拆分，避免所有逻辑继续堆积在 `App.tsx` 中。
- 为后端提供稳定接口，包括文件上传、解析结果、SSE 事件流、评分和候选人状态更新。
- 使用 Prisma 建立清晰的数据模型，持久化候选人、岗位需求、提取结果、评分结果与文件元数据。
- 通过 BigModel HTTP API 完成简历结构化提取与 JD 匹配评分，并支持流式过程反馈。

**Non-Goals:**
- 不在本次设计中实现完整用户系统、权限系统或多租户隔离。
- 不承诺生产级 PDF 解析精度和所有简历模板兼容性。
- 不实现对象存储、分布式任务队列或高可用文件服务。
- 不支持一次上传多个 PDF 文件。

## Decisions

### 1. 前端采用“单入口页面 + 业务组件分层”而不是立刻引入完整路由系统
- 决策：保留当前单页应用入口，在 `frontend/src` 下新增招聘工作台页面组件、上传区、JD 编辑器、候选人列表、详情抽屉/面板、评分图表等业务组件。
- 原因：当前项目没有路由和多页面结构，本次只新增一个核心页面，直接引入重型路由改造收益低。
- 备选方案：
  - 引入 React Router 并拆成多路由页面：更规范，但会扩大范围。
  - 继续全部写在 `App.tsx`：实现快，但后续不可维护。

### 2. 后端使用 SQLite + Prisma 作为唯一数据存储
- 决策：通过 Prisma schema 建模候选人、简历文件、解析结果、结构化信息、岗位需求、评分结果和状态流转，底层使用 SQLite 存储。
- 原因：用户明确要求落库持久化，SQLite + Prisma 能在当前原型阶段提供足够的开发效率、迁移能力和查询结构。
- 备选方案：
  - 内存态仓库：已被排除。
  - PostgreSQL：更适合生产，但会增加本地环境成本。

### 3. PDF 文件落盘到仓库根目录专用目录，数据库保存元数据
- 决策：后端接收上传后的 PDF，使用 `fs` 写入根目录固定文件夹，例如 `storage/resumes/`，Prisma 表中保存原始文件名、存储路径、大小、MIME 类型和关联候选人。
- 原因：用户明确要求使用文件系统存储，且该方式适合本地原型和预览场景。
- 备选方案：
  - 数据库存二进制：SQLite 不适合承载此类原始文件。
  - 对象存储：超出当前范围。

### 4. 上传能力限定为单文件流程
- 决策：前端上传组件一次只接受一个 PDF 文件，上传状态、解析状态、提取状态和评分状态围绕单个候选人实体推进。
- 原因：用户已明确取消批量上传需求，单文件流程能显著简化状态管理与交互复杂度。
- 备选方案：
  - 保留批量接口：与当前需求不符。

### 5. AI 能力通过 BigModel HTTP API 适配层编排
- 决策：后端定义 BigModel client/service，通过环境变量读取 API Key，按 HTTP 接口调用模型完成“简历结构化提取”和“岗位匹配评分”两个用例。
- 原因：用户已指定智谱 BigModel 文档和密钥注入方式；通过适配层可以隔离请求拼装、错误处理和后续模型替换。
- 备选方案：
  - 直接在控制器调用 HTTP：耦合高。
  - 前端直连模型：会暴露密钥且无法安全控制 SSE 流程。

### 6. SSE 采用“候选人维度事件流”而不是全局广播流
- 决策：前端在触发某份简历提取时，订阅对应候选人的 SSE 端点；后端按候选人输出阶段事件，如 `started`、`section`、`completed`、`failed`。
- 原因：候选人是天然聚合单位，便于前端局部更新，且不会让多任务流相互干扰。
- 备选方案：
  - WebSocket：能力更强，但本次只需要单向流式返回，SSE 更简单。
  - 轮询：无法满足逐步渲染要求。

### 7. 评分可视化优先选择轻量图表库
- 决策：前端使用轻量 React 图表方案展示综合分与维度分，至少包含一个综合进度图和一个多维对比图。
- 原因：用户明确要求可视化图表，同时桌面端展示需要兼顾观感和实现成本。
- 备选方案：
  - 纯 CSS 自绘：依赖少，但多维图表实现复杂。
  - 引入重量级 BI 图表库：对当前原型过重。

### 8. 状态管理采用 React 本地状态 + 领域级 hooks
- 决策：使用页面级 `useReducer`/`useState` 与自定义 hooks 组织上传、筛选、候选人详情、主题等状态。
- 原因：目前页面数量少，业务集中，避免提前引入全局状态库。
- 备选方案：
  - 全局状态库：适合更大规模应用，但本次收益有限。

## Risks / Trade-offs

- [PDF 解析库对中文或复杂排版兼容性有限] → 通过清洗规则与字段缺失兜底展示，避免把解析结果当作绝对真值。
- [BigModel 输出不稳定，字段可能缺失或格式漂移] → 后端增加结构化归一层，前端支持用户手动修正关键字段。
- [SQLite 在并发写入和复杂查询方面有限] → 当前原型以单机演示为目标，数据模型尽量保持扁平清晰。
- [文件系统落盘需要处理路径与清理问题] → 统一存储目录命名规则，并在记录删除时同步清理文件。
- [单页承载多个模块可能造成组件耦合] → 通过页面容器 + 功能面板 + 领域 hooks 拆分，避免继续集中在根组件。
- [外部模型依赖环境变量与网络条件] → 在配置缺失或调用失败时明确暴露错误，不隐藏成假成功。

## Migration Plan

- 在后端新增 Prisma 初始化、schema 和迁移文件，建立 SQLite 数据库。
- 新增招聘模块及相关 controller/service，不影响现有 `GET /` 示例接口。
- 在仓库根目录创建文件存储目录并接入上传落盘逻辑。
- 增加 BigModel API Key 环境变量读取与 HTTP client 封装。
- 前端在现有 `App.tsx` 基础上替换为招聘工作台入口，并新增组件目录与样式文件。
- 若需要回滚，可移除新增模块并恢复原始页面；SQLite 文件与上传目录可单独清理。

## Open Questions

- 详情页采用右侧抽屉还是整页展开，需在实现时结合桌面端布局决定。
- 首版是否实现多个 JD 同时对比评分，还是先完成单 JD 评分并预留扩展结构。
- 文件存储目录的具体命名是否固定为 `storage/resumes/`，还是使用用户指定目录名。
