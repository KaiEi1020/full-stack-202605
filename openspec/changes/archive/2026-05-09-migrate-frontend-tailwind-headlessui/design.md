## Context

当前前端已经从单一 GraphQL demo 扩展到上传、候选人管理、详情、注册等多个页面，样式主要集中在 `App.css` 中维护，并通过手写类名驱动布局和状态。随着页面持续增多，现有方式在样式复用、响应式约束、主题表达和交互组件一致性上成本越来越高，因此需要引入 Tailwind CSS 作为原子样式基础，并使用 Headless UI 承载交互组件行为，同时基于 `frontend-design` 重新优化页面视觉设计。

## Goals / Non-Goals

**Goals:**
- 在前端引入 Tailwind CSS 并建立新的样式入口与构建配置。
- 在前端引入 Headless UI，作为菜单、弹层、切换、对话框等交互组件的基础。
- 将当前核心页面从 `App.css` 主导迁移到 Tailwind class 主导的实现。
- 借迁移机会统一页面层级、间距、卡片、状态块和操作区视觉设计。
- 保持现有页面功能、路由和数据请求逻辑继续可用。

**Non-Goals:**
- 不在本次变更中重写后端接口或 GraphQL 数据结构。
- 不强制一次性引入完整设计系统文档站。
- 不要求本次把所有潜在未来页面一次性组件化完成。
- 不更换 Apollo Client、React 或 Vite 的核心技术栈。

## Decisions

### 1. 使用 Tailwind CSS 作为主要样式事实来源
- 决策：新增 Tailwind 配置、内容扫描与全局样式入口，逐步移除 `App.css` 承担的主要视觉职责。
- 原因：Tailwind 更适合当前页面密集演进阶段，可直接在组件内表达布局与状态，降低跨文件维护成本。
- 备选方案：继续增强现有 CSS 文件；未采用，因为页面数量已增长，样式耦合会继续扩大。

### 2. 使用 Headless UI 承载交互结构，而不是自写复杂行为组件
- 决策：将导航菜单、筛选器、候选人交互控件或确认流程优先建立在 Headless UI 组件模式上。
- 原因：Headless UI 提供无样式但可访问性较好的行为层，适合与 Tailwind 搭配。
- 备选方案：继续手写交互组件；未采用，因为一致性与无障碍成本更高。

### 3. 保持现有页面路由和功能边界不变，优先做 UI 迁移和视觉优化
- 决策：不改动页面职责划分，重点迁移 `App.tsx` 和现有页面组件的结构与样式表达。
- 原因：这样可以把风险限定在前端展示层，而不影响已跑通的数据流。
- 备选方案：同时重构路由和组件树；未采用，因为会把视觉迁移与架构重构耦合在一起。

### 4. 基于 frontend-design 统一页面设计语言
- 决策：建立新的视觉约束，包括导航层级、信息卡片、表单区域、列表区、详情区、状态反馈和操作工具栏。
- 原因：仅替换技术栈不足以改善可用性，必须同步优化页面信息层次和交互质感。
- 备选方案：只做 Tailwind 技术迁移；未采用，因为无法体现用户要求的“重新优化页面设计”。

## Risks / Trade-offs

- [迁移范围跨多个页面] → 先覆盖核心页面和共享外壳，优先抽取复用样式模式。
- [Tailwind class 导致 JSX 噪音上升] → 对重复模式提炼为小组件或常量，避免长串 className 失控。
- [Headless UI 引入后组件结构变化较大] → 只在确有交互收益的区域使用，不为迁移而迁移。
- [旧 CSS 与新 Tailwind 并存期间容易冲突] → 明确 Tailwind 为主、旧 CSS 仅作过渡，逐步收缩旧样式覆盖面。
