## Context

当前后端需求同时涉及协议层、持久化层、模块组织方式与简历领域建模的整体重构，属于跨模块的架构性调整。现状中 GraphQL 作为接口协议、Prisma 作为持久化入口、简历拆分多表的建模方式共同增加了实现与维护复杂度；而项目要求数据库继续使用 SQLite，因此重构需要在不更换数据库的前提下统一协议与 ORM 方案，并重新划分复杂模块与简单模块的实现风格。

## Goals / Non-Goals

**Goals:**
- 以 RESTful HTTP 接口替代现有 GraphQL 查询与变更入口
- 以 TypeORM 实体、仓储和迁移机制替代 Prisma schema 与 Prisma Client
- 将简历领域收敛为单表模型，支持以单记录承载完整简历字段
- 提供简历 AI 多维度评分结果，包括技能匹配度、经验相关性、教育背景契合度与综合评语
- 提供评分结果的图表化展示能力，支持前端以雷达图、柱状图或环形进度条呈现
- 为复杂模块保留 DDD 风格的边界、服务与仓储抽象
- 为简单模块采用直接的 CRUD 模块组织，降低样板代码成本
- 保持 SQLite 作为运行时数据库，不引入新的数据库产品

**Non-Goals:**
- 不在本次重构中引入新的鉴权、缓存、消息队列或搜索能力
- 不追求将所有模块统一为同一种架构风格
- 不保留 GraphQL 与 REST 双协议并存
- 不继续维护简历多表拆分模型及其兼容层

## Decisions

1. **协议层统一为 RESTful 控制器**
   - 决策：移除 GraphQL resolver、schema 与模块装配，改为 NestJS controller + service 组织 HTTP 接口。
   - 原因：项目明确要求放弃 GraphQL，且 REST 更适合简历这类直接资源建模与 CRUD 场景。
   - 备选方案：保留 GraphQL 并额外暴露 REST；不采用，因为会增加双协议维护成本并延长迁移周期。

2. **持久化层统一为 TypeORM + SQLite**
   - 决策：使用 TypeORM 实体映射 SQLite 表结构，并通过仓储或 TypeORM Repository 完成数据访问。
   - 原因：项目明确要求放弃 Prisma，但继续使用 SQLite；TypeORM 与 NestJS 集成成熟，适合 DDD 与 CRUD 混合场景。
   - 备选方案：使用原生 SQL 或其他 ORM；不采用，因为缺少统一的实体映射与模块化集成能力。

3. **简历领域采用单表模型**
   - 决策：创建单一 resume 表承载简历核心字段，将原先分散在多表的数据收敛到单表列或可序列化字段中。
   - 原因：项目明确认为当前简历拆表过多，单表更符合当前业务复杂度与开发效率目标。
   - 备选方案：继续保留多表，通过视图或聚合层简化使用；不采用，因为数据模型复杂性仍然存在。

4. **复杂模块保留 DDD，简单模块使用 CRUD**
   - 决策：按业务复杂度选择模块组织方式。涉及明确领域规则、边界与组合流程的模块继续保留分层与仓储接口；纯数据管理模块采用 controller/service/repository 的 CRUD 结构。
   - 原因：统一使用 DDD 会为简单模块引入过多样板；统一使用 CRUD 又会削弱复杂模块的边界表达。
   - 备选方案：全量 DDD 或全量 CRUD；都不采用，因为与项目要求不符。

5. **AI 打分结果采用结构化输出**
   - 决策：为简历评分定义固定结构，至少包含技能匹配度、经验相关性、教育背景契合度、总分与 AI 评语，以便前后端稳定联调与图表渲染。
   - 原因：评分结果若仅返回自由文本，将无法稳定支撑多维度展示与可视化组件。
   - 备选方案：只返回非结构化评语；不采用，因为无法满足多维度图表展示要求。

6. **迁移策略采用一次性结构替换优先**
   - 决策：在变更期内移除 Prisma 资产并建立 TypeORM 数据源、实体与迁移；若存在旧数据，则通过专门迁移脚本将简历多表数据汇总进单表。
   - 原因：继续同时维护 Prisma 与 TypeORM 会增加一致性成本。
   - 备选方案：长期并行维护两套持久化层；不采用，因为复杂度过高且没有业务价值。

## Risks / Trade-offs

- [接口破坏性变更] 前端或调用方原有 GraphQL 请求会失效 → 通过明确 REST 路由契约与迁移任务同步调整调用方
- [数据迁移丢失字段] 多表到单表聚合时可能遗漏边缘字段 → 先定义单表字段清单，再为每类旧表数据建立映射规则并补充迁移验证
- [架构风格不一致] DDD 与 CRUD 并存可能导致团队理解成本增加 → 通过模块级约束明确何时使用 DDD、何时使用 CRUD
- [TypeORM 替换成本] Prisma 相关代码与依赖移除范围较广 → 先建立最小可运行 TypeORM 基础设施，再逐模块迁移
- [SQLite 能力边界] 单表承载更多字段后，复杂查询或索引策略可能受限 → 优先按当前需求设计字段，避免过度优化，必要时补充索引

## Migration Plan

1. 识别并冻结现有 GraphQL、Prisma 与简历多表相关入口
2. 建立 TypeORM 数据源配置、SQLite 连接与基础实体映射
3. 定义 resume 单表实体与字段映射规则，准备迁移脚本或初始化迁移
4. 将简历相关接口改造为 RESTful controller/service/repository 链路
5. 重构复杂模块，保留 DDD 分层与仓储边界；简单模块调整为 CRUD 结构
6. 移除 GraphQL 模块与 Prisma 依赖，更新应用启动与模块装配
7. 通过单元测试、接口测试与迁移验证确认行为稳定
8. 如上线后需回滚，则回退到重构前版本与旧 SQLite 数据文件/备份
