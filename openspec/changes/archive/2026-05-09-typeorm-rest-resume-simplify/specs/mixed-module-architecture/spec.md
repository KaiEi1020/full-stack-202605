## ADDED Requirements

### Requirement: 系统必须按模块复杂度选择架构模式
系统 MUST 根据业务复杂度为后端模块选择 DDD 或 CRUD 架构模式，而不是对所有模块采用统一分层策略。

#### Scenario: 装配复杂业务模块
- **WHEN** 模块包含明确的领域规则、跨对象协作或稳定边界需求
- **THEN** 系统必须保留领域层、应用层与基础设施层等 DDD 组织方式

#### Scenario: 装配简单业务模块
- **WHEN** 模块主要执行直接的数据增删改查且缺少复杂领域规则
- **THEN** 系统必须允许采用 controller、service、repository 直连的 CRUD 组织方式

### Requirement: 架构模式选择不能破坏模块边界
系统 MUST 在 DDD 与 CRUD 并存的情况下保持模块职责清晰，避免复杂模块因简单模块风格而失去边界约束。

#### Scenario: 复杂模块调用数据访问实现
- **WHEN** 复杂模块执行核心业务流程
- **THEN** 系统必须继续通过定义良好的仓储边界或应用服务协作访问持久化能力

#### Scenario: 简单模块暴露接口
- **WHEN** 简单模块暴露资源接口
- **THEN** 系统必须直接围绕资源与持久化操作组织代码，而不引入无业务价值的额外领域抽象
