## MODIFIED Requirements

### Requirement: 系统必须提供 RESTful 简历接口
系统 MUST 通过标准 HTTP RESTful 接口提供简历资源的创建、查询、更新与删除能力，而不是通过 GraphQL 查询或变更暴露该能力。

#### Scenario: 创建简历
- **WHEN** 客户端向简历集合资源发送合法的创建请求
- **THEN** 系统必须创建一条新的简历记录并返回创建结果

#### Scenario: 查询简历详情
- **WHEN** 客户端使用简历标识访问单个简历资源
- **THEN** 系统必须返回对应简历的完整字段数据或明确的不存在结果

#### Scenario: 更新简历
- **WHEN** 客户端向单个简历资源发送合法的更新请求
- **THEN** 系统必须持久化变更并返回更新后的简历结果

#### Scenario: 删除简历
- **WHEN** 客户端请求删除已有简历资源
- **THEN** 系统必须删除对应记录或按约定返回删除成功结果

### Requirement: REST 接口必须返回明确的 HTTP 语义
系统 MUST 使用符合 RESTful 风格的 HTTP 方法、状态码与 JSON 响应结构表达成功、失败与资源不存在等结果。

#### Scenario: 查询不存在的简历
- **WHEN** 客户端请求不存在的简历资源
- **THEN** 系统必须返回资源不存在的 HTTP 状态，而不是进程错误或 GraphQL 风格错误包装

#### Scenario: 提交非法简历数据
- **WHEN** 客户端向简历接口提交不满足约束的请求体
- **THEN** 系统必须返回请求无效的 HTTP 状态与可识别的错误信息

#### Scenario: 状态流转展示与操作一致
- **WHEN** 用户在简历详情页查看或修改简历状态
- **THEN** 前端必须能够明确区分当前状态与可操作状态，并在状态更新后反映最新结果
