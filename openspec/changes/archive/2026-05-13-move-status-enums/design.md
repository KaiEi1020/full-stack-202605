## Context

当前后端已经按 `modules/` 与 `core/` 做顶层归类，其中 `core/database` 应承担数据库连接、配置、模块装配等基础设施职责。但现有 `CandidateStatus`、`JobStatus`、`ParseStatus` 被放在该目录下，并被实体、DTO、service、编排服务跨模块引用，导致业务语义与基础设施语义发生混用。

这次改造是一次小范围但跨模块的结构整理：不改变枚举值、不改变数据库字段存储、不改变 API 行为，只修正类型定义的归属位置与引用路径。

## Goals / Non-Goals

**Goals:**
- 将业务状态枚举迁移到更贴近业务的目录，消除 `core/database` 对业务语义的承载。
- 保持 TypeORM 实体列定义、DTO 类型声明、业务判断逻辑不变。
- 为后续新增业务状态类型建立统一归类规则：业务类型归业务模块或共享业务类型目录，数据库层只负责持久化接入。

**Non-Goals:**
- 不调整数据库表结构、列类型或枚举值。
- 不重构候选人、岗位、解析流程本身的业务逻辑。
- 不引入新的共享抽象或额外依赖。

## Decisions

### 决策一：按业务归属迁移枚举，而不是继续集中放在 `core/database`
- 选择：将 `CandidateStatus` 放入 `modules/candidate` 附近，将 `JobStatus`、`ParseStatus` 放入各自最直接的业务模块或共享业务类型目录。
- 原因：这些枚举表达的是领域状态，而不是数据库能力。数据库层可以引用它们，但不应定义它们。
- 备选方案：继续保留在 `core/database`，仅通过命名说明其业务属性。
- 不采用原因：目录语义仍然错误，后续会继续误导新增代码的归类。

### 决策二：优先做最小迁移，只调整定义位置与 import
- 选择：保留现有枚举名称、成员和值，仅迁移文件位置并更新引用。
- 原因：这次目标是修正边界，不是重塑领域模型；最小迁移更安全，也不影响数据库存量数据。
- 备选方案：顺手统一为 `shared/enums` 或拆出独立 domain 层。
- 不采用原因：会扩大改造范围，超出本次需求。

### 决策三：`core/database` 仅保留基础设施导出
- 选择：删除其中的业务状态枚举导出，避免新的 import 继续从数据库层拿业务类型。
- 原因：如果保留旧导出，会形成“文件移动了但依赖边界没修正”的假迁移。
- 备选方案：在 `core/database/index.ts` 保留兼容 re-export。
- 不采用原因：会延续错误依赖方向，不利于约束落地。

## Risks / Trade-offs

- [跨模块引用遗漏] → 通过全局搜索 `CandidateStatus`、`JobStatus`、`ParseStatus` 并执行构建或测试验证。
- [导入路径变更导致编译失败] → 迁移后先检查 TypeScript 编译与 Nest 构建，再修正残余引用。
- [枚举归属仍有争议] → 以“谁拥有该业务状态，谁定义该类型”为准；若被多个业务共享，再放入共享业务类型目录。
- [删除旧 re-export 影响面超出预期] → 若发现确有多个业务共享且短期无法收敛，再统一迁到明确的共享业务目录，而不是回退到 `core/database`。

## Migration Plan

1. 识别 `core/database` 下所有业务状态枚举及其引用点。
2. 为每个枚举选择目标目录，并移动文件。
3. 更新实体、DTO、service、orchestrator 与 barrel 导出路径。
4. 删除 `core/database` 中相关导出，保证数据库层不再暴露业务状态定义。
5. 运行构建或测试，确认迁移后行为不变。

## Open Questions

- `JobStatus` 和 `ParseStatus` 是否分别已经有明确的单一归属模块；若没有，是否应进入共享业务类型目录。
- 项目是否希望后续新增 `shared/types` 或 `shared/enums`，承载确实跨业务共享的领域状态类型。
